## Summary
The repo-local printable-target CLI already has a working `init` and `generate` path, but it still feels like an internal utility rather than the official target-generation app the backlog calls for. The biggest gaps are user-facing rather than architectural: the help surface is sparse, users cannot validate a spec without attempting a full render, and ChArUco dictionary discovery still requires reading source/docs instead of asking the CLI directly. `PRINT-004` should turn the existing CLI into the official repo-local printable-target app by adding those missing affordances, tightening the help/docs around them, and keeping the distribution story explicit: this CLI remains repo-local for now and is not being published in this task.

## Decisions Made
- Keep the CLI repo-local in this task. Do not publish `calib-targets-cli`, do not add release workflow for it, and do not change the current documented distribution stance.
- Treat the existing `init` / `generate` commands as the stable core workflow and add only the minimal affordances needed to make that workflow self-service: spec validation and dictionary discovery.
- Keep new behavior as thin wrappers over existing library logic. `validate` should reuse printable-document validation instead of re-implementing schema or geometry checks, and dictionary listing should reuse the built-in ArUco dictionary table already generated by `calib-targets-aruco`.
- Improve the CLI’s product quality through help text and docs rather than by broadening scope into packaging, new target types, or a larger app redesign.

## Risks/Open Questions
- The binary target is still named `calib-targets`, which already produces the known Cargo doc filename-collision warning against the facade lib. Renaming or repackaging the binary is outside this task and should not be mixed into CLI workflow productization.
- A `validate` command can either print nothing on success or emit a short summary. The task should choose one deterministic success output and document it; do not leave the UX ambiguous.
- The ArUco built-ins generator already emits a sorted `BUILTIN_DICTIONARY_NAMES` table. The CLI should use that stable source rather than introducing a second hard-coded dictionary list.

## Role-Specific Details

### Architect Planning
- Problem statement:
  The current printable-target CLI works, but it still requires users to infer too much from docs or source code. A user can initialize and generate, but cannot ask the CLI whether a spec is valid before rendering, and cannot discover valid ChArUco dictionary names from the CLI itself. On top of that, the help output is sparse enough that the CLI does not yet read like an official repo-local app. That makes printable generation less self-service than it should be at this stage of the backlog.
- Scope:
  Productize the repo-local printable-target CLI by adding a `validate` command for spec checking, a `list-dictionaries` command for discoverable ChArUco dictionary names, clearer clap help/arg descriptions across the printable workflow, and matching README/book/test coverage that presents this CLI as the official repo-local target-generation app.
- Out of scope:
  Publishing the CLI on crates.io, changing the binary/package naming scheme, redesigning the JSON model, adding new target families, changing print/render library behavior, Python binding changes, or wider packaging/distribution decisions beyond explicitly documenting that the CLI remains repo-local for now.
- Constraints:
  Preserve the current crate layering: the CLI remains a thin wrapper over `calib-targets-print` and related detector metadata. Keep the change additive and reviewable. Reuse existing printable validation and ArUco built-in dictionary sources. Avoid silent behavior changes in `init` and `generate`. Keep docs accurate about repo-local-only distribution.
- Assumptions:
  `PrintableTargetDocument::load_json` plus the printable validation path are already sufficient for a dedicated `validate` subcommand.
  `calib_targets_aruco::builtins::BUILTIN_DICTIONARY_NAMES` is the intended stable source for dictionary discovery output.
  The productized CLI should still be invoked through `cargo run -p calib-targets-cli -- ...` in docs for now.
- Implementation plan:
  1. Add the missing operator affordances to the CLI surface.
     Extend `crates/calib-targets-cli/src/main.rs` with a `validate` subcommand that loads and validates a printable spec without writing bundle files, and a `list-dictionaries` subcommand that prints the built-in dictionary names in stable order. Keep both commands thin wrappers over existing library data/validation. Also improve clap doc strings so top-level help and subcommand help are self-explanatory instead of mostly blank.
  2. Add focused integration tests for the productized workflow.
     Expand `crates/calib-targets-cli/tests/cli.rs` to cover the new commands and key user flows: validating a known-good spec, rejecting an invalid spec via `validate`, and listing dictionaries with at least one known built-in name present. Keep the existing `init`/`generate` coverage and add one end-to-end “init -> validate -> generate” flow if it can be done without excessive duplication.
  3. Align CLI-facing docs with the official repo-local workflow.
     Update `crates/calib-targets-cli/README.md` and the CLI section of `book/src/printable.md` so the documented workflow becomes: discover dictionary names when needed, initialize or edit a spec, validate it, then generate outputs. Be explicit that the CLI is the official repo-local app today and is still not published on crates.io.
- Acceptance criteria:
  1. The CLI exposes a `validate` command that checks a printable spec without writing output files and exits non-zero for invalid specs.
  2. The CLI exposes a `list-dictionaries` command that prints the available built-in dictionary names for ChArUco initialization in stable order.
  3. Top-level and subcommand help output is materially clearer than the current blank/minimal help, with descriptions for the printable workflow and its key arguments.
  4. The integration tests cover dictionary discovery, successful validation, failed validation, and the existing init/generate flow.
  5. The CLI README and printable guide document the productized workflow and explicitly state that the CLI remains repo-local rather than published on crates.io.
  6. The task does not change printable rendering behavior or publish/distribution mechanics beyond clearer repo docs.
